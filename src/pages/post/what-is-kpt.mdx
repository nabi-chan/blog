## KPT 회고란 무엇일까?

KPT 회고는 `Keep`, `Problem`, `Try` 로 관점을 나누어 회고를 하는 방법입니다.

- 지키고 싶은 좋은 것들은 `Keep`에,
- 문제 상황이라고 생각하는 것들은 `Problem`에,
- 시도해볼만한 구체적인 내용들은 `Try`에 넣어두는 방식입니다.

## 어떻게 하는걸까?

###### 0. 준비물

포스트잇 2개를 준비합니다. (색상은 서로 다른 것을 추천드려요)

###### 1. `Keep`, `Problem` 정의하기

이 중에서 **계속하여 지켜나가고 싶은 만족한 내용**은 `Keep` 포스트잇에,
**고치고 싶은 내용**이나, **잠재적인 문제**는 `Problem` 포스트잇에 각각 작성합니다.

###### 2. `Keep`, `Problem` 공유하기

제한시간이 모두 지나면, 각자의 `Keep`과 `Problem` 을 발표하고, 보드에 붙입니다.

###### 3. `Try` 작성하기

모든 사람의 포스트잇이 보드에 붙으면, 이제부터 `Try` 에 대해서 고민할 차례입니다.

아래 내용이 `Try` 작성에 도움이 될 수도 있습니다.

- 최대한 구체적이여야 합니다.
  - 코드 퀄리티가 아쉽다 -> 코드를 더욱 잘쓰자 (X), 코드 리뷰 시스템을 정착하자 (O)
- 다음 회고에서 측정이 가능해야 합니다.
- 자신이 할 수 있는 범위에서 정합니다.

###### 4. `Try` 공유하기

제한시간이 끝나면, `Try` 도 똑같이 발표를 하고 보드에 붙입니다.
이때, 궁금한 점이 있다면 발표가 끝나고 언제든지 물어볼 수 있습니다.

###### 5. `Action` 공유하기

위에서 나온 `Try`를 기준으로, Action Plan을 정의합니다.

만약 **'코드 리뷰 시스템을 정착하자'** 라는 `Try` 가 나온 경우, Approve는 몇명이 할 것인지, 리뷰의 범위는 어디까지인지 등을 정해야 할 것입니다.

성과로 삼을 지표도 이 단계에서 같이 설정하는것을 추천드립니다.

## 예시로 알아보는 KPT 회고

예시를 들어, 저는 이런 방식으로 이번주의 KPT 회고를 작성할 수 있습니다.

```mdx [filename="KPT_회고_예시.md"]
### Keep

- 매일 30분씩 걷기 운동 하는 것
- 블로그 글에 대한 아이디어를 계속 구상하는 것

### Problem

- 무엇인가를 충동구매 하는 것
- 할 일이 많아 무엇을 먼저 해야할지 모르는 것

### Try

- 구매 하기 전에 3일간 필요한 이유를 생각하기
- 할 일을 중요도와 시급도에 맞춰 나누기

### Action Plan

- 구매 일지 쓰기
  - 구매 예산 정하기
  - 필요한 이유 하루에 1개 이상 작성하기
- 할 일 목록 나누기
  - 노션에 할 일 목록을 작성하기
  - 시급도 / 마감일 순으로 정리하기
```

## 글을 마치며

여기까지 KPT 회고에 대해서 알아보았습니다.

큰 단위의 팀에서 진행해도 좋은 회고 방식이지만, 작은 단위 (개인 단위에서도!) 사용해보면 좋을 것 같은 방법론이라고 저는 개인적으로 생각합니다.

만약 팀에 회고가 필요한데, 방법이 없다면 이 방식을 시도해보는 것은 어떨까요?
